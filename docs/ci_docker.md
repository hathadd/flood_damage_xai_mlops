# CI and Docker Readiness

This project keeps its existing local workflow and adds two optional readiness layers:

- a lighter but more explicit CI validation path;
- container images for FastAPI and Streamlit.

These additions do not replace the current local launch mode with `uvicorn` and `streamlit`.

## Local Launch

Run the backend locally:

```bash
source .venv-wsl/bin/activate
uvicorn src.serving.app:app --host 127.0.0.1 --port 8000
```

Run the Streamlit frontend locally in a second terminal:

```bash
source .venv-wsl/bin/activate
streamlit run src/demo/streamlit_app.py
```

By default, Streamlit targets `http://127.0.0.1:8000`.

## CI Scope

The GitHub Actions CI currently performs lightweight validation:

- dependency installation with `requirements-ci.txt`;
- presence checks for critical config and serving files;
- Python compilation for `src`, `tests`, and `scripts`;
- lightweight tests for imports, FastAPI app readiness, monitoring script imports, and Docker readiness files.

This is CI, not CD:

- `CI` validates that the repository remains structurally healthy;
- `CD` would automatically package, publish, and deploy services.

No deployment pipeline is added here.

## Docker Launch

Build and run both services:

```bash
docker compose up --build
```

Services:

- FastAPI: `http://127.0.0.1:8000`
- Streamlit: `http://127.0.0.1:8501`

Stop the stack:

```bash
docker compose down
```

## Docker Design Choices

- `Dockerfile.api` packages only the FastAPI serving layer.
- `Dockerfile.streamlit` packages only the Streamlit frontend.
- the raw dataset is not copied into the images;
- large checkpoints are not baked into the images.

Instead, `docker-compose.yml` mounts only the model assets needed by serving:

- `./mlruns`
- `./mlruns_registry_artifacts`
- `./outputs/focal_run_b_regularized/checkpoints`

This keeps Docker optional and avoids turning the image into a full training artifact bundle.

The Compose API service uses `FLOOD_DAMAGE_MODEL_SOURCE=auto`:

- it first attempts MLflow Registry loading;
- if the local registry metadata is not portable inside the container, the existing serving fallback can still use the mounted checkpoint.

## API Base URL in Docker

Inside Docker Compose, Streamlit uses:

```text
http://api:8000
```

This is injected through the `FLOOD_DAMAGE_API_BASE_URL` environment variable. Local launches still default to `http://127.0.0.1:8000`.

## Limits

- No cloud deployment is configured.
- No image publishing pipeline is configured.
- No automatic model registration or promotion is triggered by CI.
- The API container still depends on mounted model assets to perform real inference.
- A local file-based MLflow Registry may not be fully portable across environments if its metadata stores host-specific absolute paths.
- Monitoring reports remain generated by the existing scripts, not by an automated deployment pipeline.
